# Master Prompt: Pixel Puzzle Maker & 3D Manufacturing Engine

**Role & Goal:** You are an expert Frontend Architect and 3D Web Graphics Specialist. Build a complete, highly polished, production-ready Single Page Application (SPA) called **"Pixel Puzzle Maker"**. This app allows users to upload a photo, downscale it, quantize it using an entirely client-side clustering algorithm, preview the layout as a 3D-printable waffle grid tray and matching tiles using Three.js, and export manufacturing-ready 3D assets (.3MF for multi-color printers or .STL/.ZIP packages for standard printers).

---

## 🛠 TECH STACK & ARCHITECTURE RULES
- **Framework:** Next.js 16 (App Router), React 19, TypeScript, **pnpm** package manager.
- **Styling:** Tailwind CSS v4 (configured globally in `app/globals.css`, no old `tailwind.config.js`).
- **UI Components:** shadcn/ui components (`button`, `slider`, `tabs`, `card`, `label`, `select`, `dropdown-menu`, `toast`).
- **3D Graphics:** Vanilla `three` with `@types/three`. Use `three/addons/controls/OrbitControls.js` for interaction and `three/addons/exporters/3MFExporter.js` / `STLExporter.js` for manufacturing pipelines.
- **Backend:** 100% Client-side. No databases, no external APIs, no server components.

## 🧭 PHASED IMPLEMENTATION DIRECTIVE

*To prevent code truncation, please build the application structure layer by layer following this exact phase sequence:*

### Phase 1: Full Interactive UI Shell & Reactive State Setup
- Build out the entire component tree structure (`page.tsx`, `control-panel.tsx`, `upload-zone.tsx`, `color-legend.tsx`).
- Initialize all state hooks for current file, selected grid size, color constraints, shapes, printer bed boundaries, and active tabs. Ensure all controls elegantly display "Disabled" placeholder states until a file is loaded.

### Phase 2: High-Performance Voxel Core Engine (`lib/image-processing.ts`)
- Implement a dedicated Web Worker structure (`lib/image.worker.ts`) to handle heavy workloads off the main UI thread.
- **Downscaler:** Canvas center-crop "cover" alignment utilizing a strict nearest-neighbor pixel extraction.
- **Deterministic Quantizer:** Implement a browser-native **k-means++** clustering algorithm. Swap out standard non-deterministic random functions with a fixed-seed Pseudo-Random Number Generator (PRNG) to guarantee that re-running the identical source image consistently returns the exact same palette and character allocations.
- Output a single-source voxel matrix data structure: `matrix[x][y] = { hexColor: string, label: string }`.

### Phase 3: Advanced Coordinate Mapping & Honeycomb Layout Engine (`lib/grid-engine.ts`)
- Map logical pixel indices `(x, y)` to real-world 3D viewport spatial coordinates `(worldX, worldY, worldZ)`.
- **Cartesian Layout:** Strict linear projection for Squares and Cylinders.
- **Honeycomb Layout:** For Hexagons, shift odd rows horizontally by exactly half a tile width, multiplying the vertical row translation step by $\sin(60^\circ)$ to nest elements perfectly flush without floating gaps.

### Phase 4: Procedural 3D Mesh & Text Generators (`lib/geometry-generators.ts`)
- Construct the 3D scene representation inside a persistent Canvas container using `OrbitControls`, directional accent casting lights, and a shadow ground projection map.
- **The Pocketed Base Tray:** A solid baseline solid box with a matrix pattern of pocket voids routed out based on the active shape profile.
- **The Interactive Tiles:** Procedurally generate matching individual tiles. To guarantee successful real-world 3D printing, apply a hard-coded $0.2\text{mm}$ offset clearance shrinkage around every tile contour face so pieces freely drop into the tray pockets.
- **3D Letter Text Integration:** Use a text geometry approach (or ultra-clean dynamic canvas alpha textures mapped to the surface meshes) to display the character strings (`A`, `B`, `C`) physically onto the floor pockets or tile faces according to the UI panel toggles.
- **Optimization Constraint:** Render the matrix utilizing **`InstancedMesh`** architectures or unified geometry buffer merges to ensure the WebGL canvas preserves a fluid 60 FPS profile even on max $128 \times 128$ configurations ($16,384$ entities).

### Phase 5: Intelligent Board Splitting & Snap-Fit Connectors (`lib/board-splitter.ts`)
- Evaluate the physical puzzle footprint against the active printer bed dimensions. If the board limits are broken, automatically divide the base tray mesh into safe sub-board quadrants.
- Prevent micro-sliver errors: Ensure edge divisions do not result in fragile segments less than 4 grid lines wide by adjusting split offsets gracefully.
- **Snap-Fit Joints:** Cut identical modular alignment slots (e.g., locking bowtie or puzzle-tab keys) cleanly along the shared connecting borders of neighboring split boards. Generate an additional standalone "Connector Peg" mesh file to bridge the pieces physically post-print.

### Phase 6: Production 3MF & ZIP Binary Exporters (`lib/exporters.ts`)
- **3MF Multi-Color Engine:** Wire the primary export to pass the colored instance array configurations to the `3MFExporter`. Materials must lock their underlying hexadecimal color values natively so slicing clients (Bambu Studio, PrusaSlicer) correctly interpret the multi-material filament assignments.
- **ZIP Packaging Engine:** Update the traditional STL pipeline. If a board requires splitting, bundle all generated sub-board files along with the connector key assets into a clean, uncompressed `.zip` archive on the fly using light client-side execution utilities. Trigger a browser file save download paired with active UI Toast notifications status alerts.
